## 1. 自我介绍话术

### 1.1 1 分钟版

```
我独立开发了一个基于 CLIP + Faiss + LLM 的多模态图像检索引擎。

三条主线:
第一, 用 CLIP 实现跨模态编码, Faiss IVF4096+PQ32x8 索引把检索速度从
    O(N) 降到 O(N/4096×nprobe), 同时 PQ 实现 64 倍向量压缩;
第二, 设计双缓冲增量索引架构——主索引 + 内存 buffer + 逻辑删除集,
    解决 Faiss IVF 不支持动态增删的问题;
第三, 通过 Function Calling 集成 LLM, 提供查询扩展和结果解释,
    LLM 不可用时自动降级为纯检索。

代码约 2800 行, 8 个核心模块, 每个模块可独立测试。
```

### 1.2 3 分钟版

```
【项目定位】
这是一个多模态图像检索引擎, 核心是"用自然语言搜图, 用图搜相似图"。

【检索引擎 — CLIP + Faiss】
CLIP (ViT-B/32) 把图像和文本编码到同一个 512 维向量空间。
Faiss 用 IVF4096+PQ32x8——IVF 倒排把搜索范围缩小到最近 nprobe 个聚类,
PQ 乘积量化把每个向量从 2048 字节压缩到 32 字节, 压缩比 64。
检索接口统一处理文本/图片路径/数组三种输入。

【增量索引 — 双缓冲架构】
这是我设计的核心。Faiss IVF 索引训练好后不支持原地增删。
我用了三件套: 主索引负责存量数据高效检索,
内存 buffer 接收新增数据做暴力搜索,
deleted_ids 集合做逻辑删除标记。
增删都是 O(1), 搜索时双路查询合并排序,
buffer 超过阈值就 merge 到主索引 (IVF add 不需要 retrain),
删除积累多了就 rebuild 物理清理。

【LLM 辅助】
通过 Function Calling 格式定义了 4 个 Agent 工具,
LLM 做查询扩展 ("狗"→"puppy/canine/金毛"→多路检索去重融合)
和结果解释, LLM 没配置时自动降级为纯检索。
```

## 2. 高频面试题

### 项目概述类

**Q: 这个项目是做什么的?**
A: 多模态图像检索引擎。用自然语言搜图, 或用图搜相似图。CLIP 编码 + Faiss 检索 + LLM 辅助增强。

**Q: 和 RAGFlow 这种项目有什么区别?**
A: RAGFlow 是生产级全模态文档理解+知识库平台, 几十万行代码, 支持 20+ 种文档格式解析、混合检索、GraphRAG、可视化 Agent 画布。我这个是聚焦向量检索底层的实验系统——Faiss 索引结构、增量更新、ANN 算法基准测试, 这些是 RAGFlow 用 Elasticsearch 帮你屏蔽掉的细节。我研究 RAGFlow 源码也是在反向学习生产级架构怎么设计。

**Q: 这是 RAG 系统吗?**
A: 实现了 RAG 中最基础的"检索+生成"部分——用 CLIP 做向量检索获取相关图片作为上下文, LLM 基于检索结果进行解释和问答。但完整的 RAG 系统还包括文档分块 (chunking)、混合检索 (BM25+向量)、重排序 (re-ranking)、多模态文档解析等能力, 这些在 RAGFlow 里都有, 我的项目还没有。所以我定位是"图像检索引擎 + LLM 辅助增强", 不是完整 RAG 系统。

---

### 检索引擎类

**Q: 为什么选 IVF4096+PQ32x8 这个参数?**
A: IVF4096: 经验公式 `4×√N` ~ `16×√N`, 12 万数据 → 1386~5543, 取 2 的幂 4096。PQ32x8: 在我的 benchmark 中 16 子段是 sweet spot, 32 子段精度略高但耗时翻倍; 8bit 在 CPU 上有 SSE 指令级优化, 比 7bit/9bit 快。组合下来 64 倍压缩, nprobe 可调精度。

**Q: 为什么图像要 L2 归一化但文本不要?**
A: CLIP 训练时文本编码器内部做了归一化, 图像编码器没有。归一化后 cos_sim = dot_product, L2 距离 = 2(1-cos_sim), 所以 Faiss 的 L2 检索结果就是语义最相似结果。

**Q: 64 倍压缩是你调出来的吗?**
A: 不是, 这是 PQ 算法的数学属性——32 段 × 8 bit = 32 bytes, 原始 512 × 4 = 2048 bytes, 2048/32 = 64。不是我优化的结果, 是选了这个参数组合的自然结果。

**Q: 为什么不用 HNSW?**
A: HNSW 查询快但内存占用高、构建慢。12 万数据 IVF+PQ 完全够用, 内存才 ~4MB。如果数据量到千万级, 会考虑 HNSW 或直接上 Milvus。

---

### 增量索引类

**Q: 双缓冲架构怎么设计的?**
A: 三件套: 主索引 (Faiss IVF) + 缓冲区 (Python list) + 删除集合 (Set)。Add 进 buffer 不动主索引, Remove 标记 deleted_ids, Search 双路查合并排序过滤, Merge buffer→主索引 add (免训), Rebuild 过滤删除物理重建。

**Q: 为什么 buffer 用暴力 dot product 而不是也建 Faiss 索引?**
A: Buffer < 1000 条 512 维, numpy dot product 就几十微秒, 不值得建索引的开销。超了就 merge 到主索引。如果要优化, 可以换 faiss.IndexFlatIP, BLAS 加速比 numpy 快 2-3 倍。

**Q: 为什么 Faiss IVF add() 不需要重新训练?**
A: 聚类中心和码本在初始 train 时就固定了。新向量只需找到最近的聚类中心放入倒排列表, 用已有码本编码。代价是新数据分布变化时编码质量下降, 所以需要定期 rebuild 用全量数据重新 train 刷新聚类中心和码本。

**Q: 逻辑删除什么时候物理清理?**
A: 调用 rebuild_index() 时——过滤 deleted_ids, 用剩余向量新建 IndexModule + 重新 train。适合在低峰期或删除积累较多时手动触发。

---

### LLM 辅助类

**Q: 查询扩展怎么做?**
A: Prompt 工程——让 LLM 从不同角度生成语义变体, temperature=0.7 保证多样性。后处理: 去重 + 确保原始查询在首位。异常时降级返回原始查询。

**Q: 什么是 Function Calling?**
A: OpenAI 定义的标准格式, LLM 输出 JSON 描述要调用哪个函数和传什么参数, 系统执行后返回结果。我的 4 个工具都按这个格式定义 Schema, LLM 根据用户意图自动选择。

**Q: LLM 不可用时怎么办?**
A: 优雅降级。`LLMInterface.available` 检查 API Key, 没有就跳过所有 LLM 功能, 退化为纯 CLIP+Faiss 检索。传统检索不需要 LLM 也能正常工作。

**Q: 查询扩展后的多路结果怎么融合?**
A: 用 Set 记录已出现的 path 去重, 保留首次出现的距离值, 全局按 distance 排序后取 topk。因为所有查询共用同一个 CLIP+Faiss, 距离值直接可比, 不需要分数归一化。

### 工程实践类

**Q: 模块是怎么划分的?**
A: 8 个模块各司其职: CLIPModel (编码)、IndexModule (Faiss 索引)、ImageRetrievalModule (检索整合)、IncrementalIndexManager (增量管理)、ImageIndexBuilder (特征提取)、LLMInterface (LLM 封装)、ImageRetrievalTools (Agent 工具)、Flask App (HTTP 层)。高内聚低耦合, 每个模块可独立测试。

**Q: 如果让你重构, 会改进哪里?**

1. 重排序: 加 Cross-Encoder 精排, 提升 top 结果准确率
2. 混合检索: BM25 关键词 + CLIP 向量双路召回
3. 多租户: 索引隔离
4. 分布式: 数据规模增大后迁移到 Milvus

## 3. 技术深度追问应对

### IVF 相关

**Q: nprobe 怎么调?**
A: 越大精度越高速度越慢。经验值 `sqrt(nlist)` 左右, 我 benchmark 过 nprobe=64 时 Recall@100 约 95%。生产中可以按场景动态调整——高精度场景用大值, 高性能场景用小值。

**Q: PQ 编码精度损失怎么评估?**
A: 我在 SIFT1M 上跑过 benchmark, 对比不同子段数 (4/8/16/32) 和 bit 数 (6/7/8/9) 的 Recall@K 曲线。PQ16x8 是 sweet spot, PQ32x8 精度最高但时间翻倍。

### CLIP 相关

**Q: CLIP 怎么训练的?**
A: 对比学习, 4 亿图文对。正样本是配对的图文, 负样本是不配对的。损失函数最大化正样本相似度、最小化负样本相似度。训练完后图文在同一向量空间。

**Q: 为什么选 ViT-B/32?**
A: B=Base (参数量适中), 32=patch size (32×32 像素一块)。输出 512 维, 比 ViT-L 的 768 维更紧凑。在检索场景已经够用, 不需要更大的模型。

### 增量索引相关

**Q: merge 后主索引质量会下降吗?**
A: 如果新数据和旧数据分布相同, 几乎没影响。如果分布变了, PQ 编码质量下降。定期 rebuild 可以彻底解决——用全量数据重新 train, 刷新聚类中心和码本。

**Q: 搜索时 buffer 太大怎么办?**
A: 设置 buffer_size_threshold (默认 1000), 超了就建议 merge。这是设计上的自觉——趁 buffer 还没大到影响搜索性能就合并到主索引。
