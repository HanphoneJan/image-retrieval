## 4. 关键设计决策

### 4.1 为什么选CLIP而不是传统CNN？

```
┌─────────────────────────────────────────────────────────────┐
│                    方案对比                                  │
├──────────────────┬────────────────────┬─────────────────────┤
│       维度        │   传统CNN (ResNet) │       CLIP          │
├──────────────────┼────────────────────┼─────────────────────┤
│ 特征空间         │ 图像专用           │ 图像+文本统一空间    │
├──────────────────┼────────────────────┼─────────────────────┤
│ 检索能力         │ 只能以图搜图        │ 以文搜图+以图搜图    │
├──────────────────┼────────────────────┼─────────────────────┤
│ 语义理解         │ 需标注训练          │ 零样本能力          │
├──────────────────┼────────────────────┼─────────────────────┤
│ 特征维度         │ 2048维             │ 512维 (更紧凑)      │
├──────────────────┼────────────────────┼─────────────────────┤
│ 预训练数据       │ ImageNet (分类)     │ 4亿图文对 (对比学习) │
└──────────────────┴────────────────────┴─────────────────────┘
```

**核心原因**:
1. **跨模态统一空间**: CLIP将图像和文本编码到同一512维空间，使"狗的图片"和"a dog"向量相近
2. **零样本能力**: 无需针对特定领域重新训练，开箱即用
3. **语义级匹配**: 基于对比学习，理解语义而非仅视觉特征

### 4.2 为什么选IVF-PQ索引？

```
┌──────────────────────────────────────────────────────────────┐
│                    Faiss索引类型对比                          │
├─────────────┬─────────────┬─────────────┬────────────────────┤
│   索引类型   │   原理      │   优势      │    适用场景        │
├─────────────┼─────────────┼─────────────┼────────────────────┤
│   Flat      │  暴力搜索   │  精度100%   │  < 10K数据         │
│             │  O(N)遍历   │             │                    │
├─────────────┼─────────────┼─────────────┼────────────────────┤
│   IVF       │  倒排索引   │  加速10-100x│  中大规模          │
│             │  k-means聚类│  内存友好   │                    │
├─────────────┼─────────────┼─────────────┼────────────────────┤
│   PQ        │  乘积量化   │  压缩127x   │  大规模存储        │
│             │  子向量编码 │             │                    │
├─────────────┼─────────────┼─────────────┼────────────────────┤
│   IVF+PQ    │  组合优化   │  速度+压缩  │  ✅ 百万级推荐     │
│             │             │  平衡       │                    │
└─────────────┴─────────────┴─────────────┴────────────────────┘
```

**数据规模计算** (COCO数据集 ~12万张):

```python
# Flat索引内存
120K × 512维 × 4字节 = 240MB
搜索复杂度: O(N) = 120K次距离计算

# IVF4096,PQ32x8
- 倒排列表: 4096个聚类中心，每类约30个向量
- PQ压缩: 每个向量 512/32=16个码，每码8bit = 16字节
- 总内存: 120K × 16B = 1.9MB (压缩比 ~127x)
- 搜索: nprobe个中心 + 量化距离计算
```

### 4.3 为什么做L2归一化？

**数学原理**:

```
CLIP输出未归一化的向量，但内积与余弦相似度的关系：

cos_sim(A, B) = (A·B) / (||A|| × ||B||)

如果A和B都是单位向量（L2归一化后）：
cos_sim(A, B) = A·B

Faiss的L2距离：
||A - B||² = ||A||² + ||B||² - 2(A·B)
           = 1 + 1 - 2(A·B)  [归一化后]
           = 2 - 2cos_sim
           = 2(1 - cos_sim)

结论: L2归一化后，L2距离与余弦相似度单调相关
      距离越小 → 相似度越高
```

**代码实现** (retrieval_by_faiss.py:112):

```python
# 图像编码后必须归一化
img_feat_vec /= img_feat_vec.norm(dim=-1, keepdim=True)

# 文本编码不需要（CLIP内部已处理）
# feat_text /= feat_text.norm(dim=-1, keepdim=True)  # 不需要！
```

## 6. 增量索引架构设计

### 6.1 为什么需要增量索引？

传统Faiss IVF索引的问题是：
- **添加单张图片**: 需要重新训练整个索引（2-4小时/12万图片）
- **删除图片**: IVF不支持直接删除，必须重建索引

**解决方案**: 基于缓冲区的增量索引管理

### 6.2 增量索引架构

```
┌─────────────────────────────────────────────────────────────┐
│                IncrementalIndexManager                       │
│  ┌─────────────────┐  ┌─────────────────┐                   │
│  │   Main Index    │  │  Buffer Index   │                   │
│  │  (Faiss IVF)    │  │  (Memory List)  │                   │
│  │  - 已训练数据   │  │  - 新添加数据   │                   │
│  │  - 大规模       │  │  - 小规模       │                   │
│  └────────┬────────┘  └────────┬────────┘                   │
│           │                    │                            │
│           └────────┬───────────┘                            │
│                    ↓                                        │
│           ┌─────────────────┐                               │
│           │  Search Merger  │                               │
│           │ (合并+过滤删除) │                               │
│           └─────────────────┘                               │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 核心设计决策

| 设计点 | 决策 | 理由 |
|:---|:---|:---|
| **添加策略** | 缓冲区累积 | Faiss IVF支持add()，避免频繁重建 |
| **删除策略** | 逻辑标记 | IVF不支持delete，重建成本太高 |
| **搜索策略** | 双路合并 | 主索引+缓冲区分别搜索，合并结果 |
| **持久化** | JSON+Pickle | 状态JSON可读，特征Pickle高效 |
